## 5. Operational gotchas LEARNED during Phase 6

These bit me during the import. Future agents working on similar
data-migration tasks should know:

1. **PG transaction poison.** Once any statement errors inside `BEGIN`,
   every subsequent query on that connection returns "current transaction
   is aborted, commands ignored until end of transaction block" — even
   if your inner try/catch swallows the original error. ROLLBACK or
   COMMIT (PG silently rolls back) is the only escape. For bulk inserts
   where some rows may fail: either use autocommit + per-row idempotency
   (slug pre-check), or wrap each row in SAVEPOINT/ROLLBACK TO. The
   import script chose autocommit + slug pre-check — simpler, safer,
   re-runnable.

2. **`import.meta.dirname` is empty under tsx in this environment.**
   Use `path.dirname(fileURLToPath(import.meta.url))` instead. The
   script silently picked up wrong paths (root `/.env`) for the first
   wipe attempt; nothing crashed because some prior process had loaded
   env vars. Always anchor paths to a derived `SCRIPT_DIR`.

3. **Next.js prod fetch cache survives DB wipes.** After any seed
   change, `rm -rf apps/web/.next/cache/fetch-cache && pm2 restart
   zhic-web` is mandatory. Otherwise routes serve stale data and old
   404s — confused me twice during Phase 6.3/6.4 smoke tests.

4. **Legacy `piece_type` enum still on products table.** Per locked
   decision, Categories now do the job — but the column with its
   limited enum (bed/closet/dresser/mirror/desk/bookcase/…) still
   exists. Set `piece_type` to NULL on every product insert; the
   `categoryIds` relation carries the real piece-type info. A follow-up
   migration should drop the column entirely.

5. **`products.base_price_rials` is NOT NULL.** Xlsx has many
   inquiry-only rows with no price; the import sets these to `0` and
   relies on `availability = made_to_order` + `status = draft` to
   keep them hidden until the operator fills in a price.

6. **`products.design_id` is NOT NULL.** Series-less rows (`-`) can't
   be imported. D3 = defer; operator handles later.

7. **Media `filename` collisions.** 14 of 584 .webp files share their
   base name with another file in a different folder. The import
   script disambiguates by prefixing with `<folder>[-<subfolder>]-`.
   The `byFilename` lookup table uses the safe (prefixed) name; the
   `byOriginalFilename` fallback in the variant phase picks first-match
   on collisions. Worth a `WARNING:` log if a variant lookup misses
   on a collision name.

8. **`finishFromMedia()` heuristic vs xlsx attribute column.** The
   xlsx products sheet has NO finish column. D4 = derive from filename
   (`-cream-`, `-gray-`, `-green-`, folder=`parla-2staged`). Variants
   get a synthetic `finish` axis added to whatever xlsx attributes say.
   Per-image precision is best-effort; operator audits via admin.

9. **hasMany-text fields go in `<collection>_texts`, not
   `<collection>_<fieldname>`.** Payload v3 reads `allowed_axes` (and
   any other text-hasMany on a collection) from a UNIFIED
   `<collection>_texts` table with a `path` column to distinguish
   fields, NOT from `<collection>_<fieldname>`. The sub-project D
   migration `20260521_180000_add_category_hub_fields.ts` created
   `categories_allowed_axes` and my Phase 6.3 script wrote there —
   both silently ignored on read. Symptom: PDP variant picker renders
   empty axis groups despite the DB having data. Fix lands in commit
   `736b80d`. Note: `designs_occupancies` (hasMany SELECT) DOES use
   the `<collection>_<fieldname>` pattern — different field types
   route differently. Always confirm against
   `information_schema.columns` for `<collection>_texts` before
   inserting.

### Variant cross-axis incompatibility edge case

When a variant's axes are heterogeneous within a product (e.g.
`parla-double-bed` has size-only variants AND size+finish variants),
selecting an incompatible combination (size:180 + finish:cream when
no such variant exists) resolves to `null` and the hero defaults to
the product cover image. Acceptable for now — the picker disables
chips that would lead to discontinued variants, but doesn't disable
chips for non-existent combinations. A future enhancement: pre-
compute the matrix of (axis-value) → reachable-set and disable
chips that lead to dead ends.

### Other axis Persian translations

If new axis keys appear in the data (beyond the 12 mapped today),
their labels will fall back to the raw English key in the PDP picker.
Both `apps/web/src/lib/variant-helpers.ts` (AXIS_LABEL +
VALUE_LABEL) and `services/api/scripts/import-catalog.mts`
(AXIS_KEY_PERSIAN + AXIS_VALUE_PERSIAN) need to be updated in
lockstep when adding mappings — filed earlier as a TODO to extract
to a shared `@zhic/locale` module.
