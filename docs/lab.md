# The Lab

`/lab` is the experimentation surface that lives inside the production
Next.js app. It exists so we can try motion, type, color, and 3D ideas
in real conditions — same framework, same fonts, same Lenis, same
hosting — without polluting the public site or the design system.

## Hard rules

1. **Never indexed.** The lab layout sets `robots: { index: false,
   follow: false, nocache: true }` on every child route, and the
   production `app/robots.ts` (Phase 1) explicitly disallows `/lab`,
   `/admin`, `/api`, and `/preview`.
2. **Not linked from public navigation.** The header / footer never
   reference `/lab`. The only entry points are the URL bar and a
   bookmarked link.
3. **Nothing in `/lab` is production-quality.** Code here is allowed to
   be ugly, fast-and-loose, and short-lived.
4. **Promotion path is one-way.** When an experiment earns its place,
   the patterns and tokens move into `docs/design-system.md` first, then
   the implementation moves into `src/components/`. The original `/lab`
   page can stay as a reference or be deleted.
5. **Performance is still measured.** Even in the lab, we measure
   Lighthouse / CWV on each experiment so the design system never
   inherits a regression we did not see coming.

## Current routes

```
/lab               Index of all experiments
/lab/motion        Easings, durations, reveal patterns, scroll choreography
/lab/type          Display vs body pairings, scales, masked reveals
/lab/color         Palette comparisons, contrast pairs, surface treatments
/lab/three         glTF viewer choices, draco/KTX2 budgets, AR placement
```

Add a new experiment by creating `src/app/lab/<slug>/page.tsx`. The
shared `src/app/lab/layout.tsx` automatically applies the noindex
metadata, the lab header, and the footer.

## Workflow

1. Open a question in the brand-spec discussion (e.g. "Cormorant 300 vs
   400 at display sizes").
2. Add a section to the relevant lab page that demonstrates the options
   side-by-side.
3. Measure: visual review with stakeholders, **plus** Lighthouse +
   real-device CWV check.
4. Decide. Write the decision and the measurement into the related
   `docs/design-system.md` section as a "Decision (yyyy-mm-dd)" note.
5. Promote the chosen pattern out of `/lab` into a real component or
   token.
6. Delete or archive the lab page when it has done its job.

## Robots & SEO safety

- Layout-level `noindex, nofollow, nocache, noimageindex`.
- `app/robots.ts` (built in Phase 1) explicitly disallows `/lab/`.
- The sitemap generator skips any route matching `^/lab(/|$)`.
- Lab pages must not embed real production analytics.
- Lab pages must not collect form submissions.

## What the lab is NOT

- Not a staging environment for production content. That role is filled
  by the CMS preview links generated by Payload (Phase 3).
- Not a Storybook replacement. Storybook (Phase 1) handles per-component
  states and visual regression. The lab handles cross-component
  experiments where the question is "which whole approach feels right."
- Not a personal playground that lives forever. Old experiments get
  cleaned up.
